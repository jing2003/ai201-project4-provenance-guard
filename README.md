# Project 4: Provenance Guard

## Project Overview

**Provenance Guard** is a backend API system that a creative sharing platform could plug into to classify submitted content, score confidence in that classification, show a plain-language transparency label to users, and allow creators to appeal if they believe their work was misclassified.

The system is designed around transparency instead of final judgment. It does not claim to perfectly prove whether something was written by a human or generated by AI. Instead, it combines multiple detection signals, explains the result with a user-facing label, stores each decision in a structured audit log, and keeps an appeals workflow available for creators.

## Architecture

### Submission Flow

```txt
Creator / Platform
        |
        | POST /submit
        | { text, creator_id }
        v
Flask API
        |
        | raw text
        v
Detection Pipeline
        |
        | Signal 1: Groq LLM assessment
        | Signal 2: Stylometric heuristics
        | Signal 3: Formulaic/repetition heuristics
        v
Confidence Scoring
        |
        | combined_ai_score + attribution + confidence
        v
Transparency Label Generator
        |
        | plain-language label
        v
SQLite Database + Audit Log
        |
        | stored classification event
        v
JSON Response
```

### Appeal Flow

```txt
Creator / Platform
        |
        | POST /appeal
        | { content_id, creator_reasoning }
        v
Flask API
        |
        | lookup content_id
        v
Status Update
        |
        | status becomes under_review
        v
Appeal Record + Audit Log
        |
        | creator reasoning stored
        v
JSON Response
```

The submission flow classifies content, combines signals, creates a transparency label, stores the decision, and returns JSON. The appeal flow does not automatically reclassify content. Instead, it marks the content as `under_review` and logs the creator's reasoning for a human reviewer.

## Detection Signals

This project uses three detection signals. Each signal outputs a score from `0.0` to `1.0`, where a higher score means the text appears more AI-like.

### Signal 1: Groq LLM Assessment

The Groq signal asks `llama-3.3-70b-versatile` to judge whether the text appears more AI-generated or human-written. It looks at broad language patterns such as polished structure, generic wording, repetitive transitions, lack of personal detail, and overall tone. I chose this signal because it can evaluate the overall tone and structure of a submission in a way that simple rules cannot.

**What it captures:**
This signal captures high-level writing patterns that are hard to express with simple rules.

**What it misses:**
It can over-trust polished writing as human or misread formal human writing as AI-generated. It may also vary depending on the model response.

### Signal 2: Stylometric Heuristics

The stylometric signal uses pure Python to measure writing-style statistics, including:

- average sentence length
- sentence length variance
- type-token ratio
- punctuation density
- casual marker count

I chose this signal because it gives explainable measurements that can be shown directly in the API response and audit log.

**What it captures:**
This signal captures measurable writing patterns. For example, highly uniform sentence structure may look more AI-like, while irregular casual writing may look more human-like.

**What it misses:**
It does not understand meaning. It may misclassify short text, formal human writing, poetry, or edited writing because the statistics may not reflect authorship.

### Signal 3: Formulaic/Repetition Heuristics

The formulaic signal checks for common generic phrases often found in AI-style writing, such as:

- `it is important to note`
- `furthermore`
- `overall`
- `paradigm shift`
- `stakeholders`
- `ethical implications`
- `responsible deployment`

I chose this signal because generic transition phrases and repeated polished wording are common patterns in AI-style text, and this signal is simple to explain to users.

**What it captures:**
This signal captures formulaic transitions and generic polished wording.

**What it misses:**
It is easy to avoid if AI text is edited. It can also over-score formal student essays or professional writing that naturally uses transitions.

## Ensemble Detection Strategy

The three signals are combined with weighted averaging:

```txt
combined_ai_score =
    0.50 * groq_score
  + 0.30 * stylometric_score
  + 0.20 * formulaic_score
```

Groq has the highest weight because it can judge broader context and overall writing style. Stylometric heuristics have the second-highest weight because they provide explainable measurements. The formulaic signal has the lowest weight because it is useful but easier to fool and more likely to overreact to formal writing.

When the signals disagree, the weighted score moves toward the uncertain range instead of trusting only one signal.

## Confidence Scoring and Uncertainty

The system separates `combined_ai_score` from `confidence`.

- `combined_ai_score` means how AI-like the content appears.
- `confidence` means how strongly the system believes the final label.

### Score Thresholds

| Combined AI Score | Attribution    |
| ----------------: | -------------- |
|       `0.00–0.30` | `likely_human` |
|       `0.31–0.69` | `uncertain`    |
|       `0.70–1.00` | `likely_ai`    |

### Confidence Calculation

For `likely_ai`, the confidence is the combined AI score.

```txt
confidence = combined_ai_score
```

For `likely_human`, confidence is based on how far the score is from AI-like.

```txt
confidence = 1 - combined_ai_score
```

For `uncertain`, confidence stays lower because the system is explicitly saying the result is unclear.

```txt
confidence = 0.40 + abs(combined_ai_score - 0.50)
```

## Confidence Validation

I tested the scoring system with deliberately different inputs to confirm that the scores vary meaningfully.

| Test Input Type                 | Attribution    | Combined AI Score | Confidence |

| Casual human-style ramen review | `likely_human` |            `0.16` |     `0.84` |
| Mixed remote-work paragraph     | `uncertain`    |            `0.38` |     `0.52` |

This showed that the system was not returning the same score for every input. The casual, irregular human-style writing produced a low AI score. ## Transparency Labels

The API returns a plain-language label with every classification result. The label text changes based on the attribution category.

| Attribution      | Label Text                                                                                                                             |
| ---------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| `likely_ai`      | `"This content appears likely to be AI-generated. This label is based on writing-pattern signals and may be appealed by the creator."` |
| `uncertain`      | `"We are not confident enough to say whether this was written by a person or generated with AI."`                                      |
| `likely_human`   | `"This content appears likely to be written by a person. No AI-generated label is being shown."`                                       |
| `verified_human` | `"Verified creator: this creator completed a provenance check for this content."`                                                      |

The labels avoid technical terms like classifier, logit, or ensemble output. A non-technical user should be able to understand what the result means.

## Rate Limiting

The `/submit` endpoint uses Flask-Limiter.

Chosen limit:

```txt
10 submissions per minute
100 submissions per day
```

I chose this limit because a normal creator may submit several drafts in a short period, but a script sending many rapid requests could abuse the API or use up the Groq free tier. The minute limit allows normal testing and small bursts of creator activity, while still blocking request flooding. The daily limit prevents extreme repeated use.

For local development, I used:

```txt
storage_uri="memory://"
```

This is acceptable for the project because local rate-limit state can reset when the server restarts.

## Stretch Feature: Ensemble Detection

The project implements three distinct detection signals:

1. Groq LLM assessment
2. Stylometric heuristics
3. Formulaic/repetition heuristics

The response shows the individual signal scores alongside the combined score:

```json
{
  "combined_ai_score": 0.77,
  "signals": {
    "groq_score": 0.8,
    "stylometric_score": 0.55,
    "formulaic_score": 1.0
  }
}
```

The documented weighting strategy is:

```txt
50% Groq
30% stylometric heuristics
20% formulaic/repetition heuristics
```

Conflicts between signals are resolved through the weighted average. A disagreement between the signals moves the final result closer to the uncertain range.

## Known Limitations

### Formal Human Writing

Formal human writing can be difficult to classify. For example, an academic paragraph may have smooth sentence structure, domain-specific vocabulary, and low casual language. The stylometric signal may treat that as AI-like, while the Groq signal may interpret it as professional human writing. This means formal essays, research summaries, or professional announcements may not always receive the expected result.

### Short Text

Short text gives the stylometric signal less information to measure. A one-sentence caption may not have enough sentence variation, punctuation patterns, or word variety for the heuristics to be meaningful.

### Poetry and Creative Writing

Poetry or experimental writing may use repetition, unusual punctuation, or simple vocabulary on purpose. The formulaic/repetition signal could misread intentional creative repetition as AI-like repetition.

### Human-Edited AI Text

If AI-generated text is heavily edited by a person, it may lose formulaic phrases and gain more personal style. The system may classify it as uncertain or likely human even though AI was involved.

### Metadata-Based Multi-Modal Support

The multi-modal endpoint does not analyze image pixels. It only analyzes descriptions, captions, and metadata. A real image provenance system would need actual image analysis, file metadata inspection, or creator-side signing.

## Spec Reflection

The planning document helped guide the implementation because it defined the three score ranges before the code was written: `likely_human`, `uncertain`, and `likely_ai`. This made it easier to test the system because I knew what kind of score each example should produce. The architecture diagram also helped separate the project into clear parts: submission, signals, scoring, labels, logging, and appeals.

One way the implementation diverged from the original plan was the audit log. At first, the log only stored scores and IDs. During testing, I realized that it was hard to understand which submitted text produced which score. I updated the `/log` endpoint to join audit log entries with the stored content so the log includes `submitted_text`. This made the audit log much easier to read and more useful for the README and demo.

Another small change was the stylometric scoring. The first version gave a very high AI-like score to a short poetic human paragraph because the sentence lengths were too consistent. I adjusted the heuristic so short text avoids extreme stylometric scores and personal markers reduce the AI-likelihood score. This made the scoring behavior better match the intended uncertainty design.

## Final Notes

Provenance Guard demonstrates a backend moderation-style system with explainable classification, uncertainty, transparency labels, appeals, rate limiting, audit logging, and stretch features. The most important design choice was treating classification as uncertain and appealable instead of final. This makes the system more realistic for a creative platform, where false positives can harm creators.
