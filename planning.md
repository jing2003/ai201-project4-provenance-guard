# Provenance Guard — planning.md

## Project Overview

Provenance Guard is a backend API system that creative sharing platforms could plug into when creators submit content. The system accepts submitted content, analyzes it using multiple detection signals, combines those signals into a confidence score, returns a plain-language transparency label, stores the decision in an audit log, and allows creators to appeal if they believe their work was misclassified.

The goal is not to perfectly determine whether content was written by a human or generated by AI. Instead, the goal is to make a transparent, explainable, and appealable moderation-style system that avoids treating automated classification as final truth.

## Core User Flow

A creator submits content to the API through `POST /submit`. The API receives the raw text and creator ID, then sends the text through a multi-signal detection pipeline. Each signal produces an AI-likelihood score from `0.0` to `1.0`, where `0.0` means strongly human-like and `1.0` means strongly AI-like.

The system combines the individual signal scores into one ensemble AI score. That score is mapped to an attribution result: `likely_ai`, `uncertain`, or `likely_human`. The system then generates a confidence score and a plain-language transparency label. The classification result, signal scores, confidence, timestamp, and content status are written to the audit log. Finally, the API returns a structured JSON response to the platform.

If a creator disagrees with the label, they can submit an appeal through `POST /appeal`. The appeal includes the original `content_id` and the creator's reasoning. The system updates the content status to `under_review`, stores the appeal, writes an appeal event to the audit log, and returns a confirmation response.

## Architecture

### Submission Flow

```txt
Creator / Platform
        |
        | POST /submit
        | { text, creator_id }
        v
Flask API
        |
        | raw text
        v
Detection Pipeline
        |
        | Signal 1: Groq LLM AI-likelihood score
        | Signal 2: Stylometric heuristic score
        | Signal 3: Formulaic/repetition score
        v
Confidence Scoring
        |
        | combined_ai_score + attribution + confidence
        v
Transparency Label Generator
        |
        | plain-language label text
        v
SQLite Database + Audit Log
        |
        | stored classification entry
        v
JSON Response
```

### Appeal Flow

```txt
Creator / Platform
        |
        | POST /appeal
        | { content_id, creator_reasoning }
        v
Flask API
        |
        | lookup content_id
        v
Status Update
        |
        | status changed from classified to under_review
        v
Appeal Record
        |
        | creator reasoning stored
        v
Audit Log
        |
        | appeal event written with timestamp
        v
JSON Response
```

The submission flow moves content through detection, scoring, labeling, logging, and response generation. The appeal flow does not automatically reclassify the content; instead, it preserves the creator's reasoning and changes the content status so a human reviewer could inspect it later.

## Detection Signals

The system will use three detection signals so the final project can qualify for the Ensemble Detection stretch feature. Each signal outputs a score from `0.0` to `1.0`, where higher means more AI-like.

### Signal 1: Groq LLM Assessment

**Tool:** Groq using `llama-3.3-70b-versatile`

**What it measures:**
This signal asks an LLM to assess whether the submitted text appears more likely to be AI-generated or human-written. It looks for broad writing patterns such as overly polished structure, generic phrasing, repetitive transitions, lack of personal specificity, and unnatural consistency.

**Output:**

```json
{
  "score": 0.0,
  "reasoning": "Brief explanation of the judgment."
}
```

The `score` is an AI-likelihood score from `0.0` to `1.0`.

**Why this may differ between human and AI writing:**
AI-generated writing often has a smooth, balanced, and generic style. It may use repeated transitions, safe explanations, and polished sentence structure. Human writing often has more irregularity, personal detail, abrupt transitions, typos, or stronger voice.

**What it misses:**
The LLM signal can be biased or inconsistent. It may over-label formal human writing as AI-generated, especially essays, professional writing, non-native English writing, or carefully edited work. It may also fail to detect AI-generated text that has been heavily edited by a person.

### Signal 2: Stylometric Heuristics

**Tool:** Pure Python

**What it measures:**
This signal measures visible writing-style statistics, including sentence length variation, word variety, punctuation usage, and casual language markers. It does not understand meaning. It only checks patterns in the text.

**Planned metrics:**

- Average sentence length
- Sentence length variance
- Type-token ratio, meaning unique words divided by total words
- Punctuation density
- Casual marker count, such as contractions, first-person phrases, and informal punctuation

**Output:**

```json
{
  "score": 0.0,
  "metrics": {
    "avg_sentence_length": 0,
    "sentence_length_variance": 0,
    "type_token_ratio": 0,
    "punctuation_density": 0
  }
}
```

The final `score` is an AI-likelihood score from `0.0` to `1.0`.

**Why this may differ between human and AI writing:**
AI-generated writing often has more even sentence lengths, smoother structure, and a moderate level of word variety. Human writing may have more uneven sentence lengths, fragments, slang, personal phrasing, repeated words, and inconsistent punctuation.

**What it misses:**
Stylometric heuristics can misclassify formal human writing as AI-like. A polished essay, academic paragraph, or professional announcement may look smooth and consistent even if it was written by a person. Short submissions are also difficult because there may not be enough text to calculate meaningful statistics.

### Signal 3: Formulaic and Repetition Signal

**Tool:** Pure Python

**What it measures:**
This signal checks for formulaic phrases and repeated patterns commonly seen in AI-generated text. Examples include phrases like "it is important to note," "in conclusion," "furthermore," "overall," "plays a crucial role," and "in today's world."

**Output:**

```json
{
  "score": 0.0,
  "matched_phrases": ["it is important to note", "furthermore"],
  "phrase_density": 0.0
}
```

The final `score` is an AI-likelihood score from `0.0` to `1.0`.

**Why this may differ between human and AI writing:**
Many AI-generated responses use generic transitions and safe explanatory phrases. A high density of these phrases may suggest AI involvement, especially when combined with smooth structure and low personal specificity.

**What it misses:**
This signal is easy to avoid if the AI output is edited. It may also misclassify student essays or formal writing that naturally uses transition words. It should never be used alone.

## Ensemble Detection Strategy

The three signals will be combined using weighted averaging:

```txt
combined_ai_score =
    0.50 * groq_score
  + 0.30 * stylometric_score
  + 0.20 * formulaic_score
```

Groq receives the highest weight because it can evaluate broader language patterns and context. Stylometric heuristics receive the second-highest weight because they provide explainable measurements. The formulaic signal receives the lowest weight because it is useful but easier to fool and more likely to overreact to formal writing.

If the signals conflict, the system will not simply follow one signal. The combined score will reflect the disagreement. For example, if the Groq signal says the text is AI-like but the stylometric score is low, the final score should move closer to the uncertain range instead of producing a high-confidence result.

## Uncertainty Representation

The system separates `combined_ai_score` from `confidence`.

- `combined_ai_score` means how AI-like the text appears.
- `confidence` means how confident the system is in the final attribution label.

### Attribution Thresholds

| Combined AI Score | Attribution    |
| ----------------: | -------------- |
|       `0.00–0.30` | `likely_human` |
|       `0.31–0.69` | `uncertain`    |
|       `0.70–1.00` | `likely_ai`    |

### Confidence Calculation

For `likely_ai`, confidence is close to the combined AI score:

```txt
confidence = combined_ai_score
```

For `likely_human`, confidence is based on how far the score is from AI-like:

```txt
confidence = 1 - combined_ai_score
```

For `uncertain`, confidence stays lower because the system is explicitly saying the result is not clear:

```txt
confidence = 0.40 + abs(combined_ai_score - 0.50)
```

Examples:

| Combined AI Score | Attribution    | Confidence | Meaning                                                 |
| ----------------: | -------------- | ---------: | ------------------------------------------------------- |
|            `0.86` | `likely_ai`    |     `0.86` | Strong AI-like pattern                                  |
|            `0.52` | `uncertain`    |     `0.42` | Too close to the middle                                 |
|            `0.23` | `likely_human` |     `0.77` | Strong human-like pattern                               |
|            `0.67` | `uncertain`    |     `0.57` | Some AI-like signals, but not enough for a strong label |

A score of `0.60` means the text has some AI-like signals, but not enough to label it as likely AI. It should receive an uncertain label with moderate-low confidence.

## Transparency Label Design

The label must use plain language and must change based on confidence and attribution.

| Attribution      | Label Text                                                                                                                             |
| ---------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| `likely_ai`      | `"This content appears likely to be AI-generated. This label is based on writing-pattern signals and may be appealed by the creator."` |
| `uncertain`      | `"We are not confident enough to say whether this was written by a person or generated with AI."`                                      |
| `likely_human`   | `"This content appears likely to be written by a person. No AI-generated label is being shown."`                                       |
| `verified_human` | `"Verified creator: this creator completed a provenance check for this content."`                                                      |

The label avoids technical terms like "classifier," "logit," or "ensemble output." A non-technical user should be able to understand the result.

## Appeals Workflow

Any creator with a valid `content_id` can submit an appeal. The appeal requires:

- `content_id`
- `creator_reasoning`

Optional future fields could include supporting evidence, draft history, or a link to prior work, but those are not required for this project.

When an appeal is received:

1. The API checks whether the `content_id` exists.
2. The content status changes from `classified` to `under_review`.
3. The creator's reasoning is stored in the appeals table.
4. An appeal event is written to the audit log.
5. The API returns a confirmation message.

A human reviewer opening the appeal queue would see:

- Content ID
- Creator ID
- Original attribution
- Original confidence score
- Individual signal scores
- Original transparency label
- Creator appeal reasoning
- Current status: `under_review`
- Timestamp of the original classification
- Timestamp of the appeal

The system does not automatically remove the label after an appeal. It only marks the content as under review so a human can make the final decision.

## False Positive Scenario

A false positive happens when a human-written post is labeled as likely AI. For example, a non-native English speaker might write in a formal and consistent style. The Groq signal might see the text as polished, the stylometric signal might detect even sentence structure, and the formulaic signal might detect formal transitions.

To reduce harm, the system has an uncertain middle range instead of forcing every result into AI or human. If the evidence is mixed, the system should return `uncertain` instead of `likely_ai`. If the result is still `likely_ai`, the transparency label clearly says the label is based on writing-pattern signals and may be appealed. The creator can then submit reasoning through `POST /appeal`, and the audit log will preserve both the original classification and the appeal.

## Anticipated Edge Cases

### Edge Case 1: Formal Human Writing

A human-written academic paragraph, professional announcement, or polished essay may be misclassified as AI-generated. This can happen because all three signals may treat formal structure, balanced sentences, and generic transitions as AI-like.

### Edge Case 2: Short Text

Very short text, such as a one-sentence caption, may not provide enough data for stylometric analysis. Sentence length variance and type-token ratio are less meaningful when there are only a few words.

### Edge Case 3: Human-Edited AI Text

If a creator heavily edits AI-generated text, the formulaic phrase signal may become weak and the stylometric signal may look more human. The system may return `uncertain` or `likely_human` even if AI was involved.

### Edge Case 4: Creative or Poetic Writing

Poetry, lyrics, or experimental prose may have repetition, unusual punctuation, or simple vocabulary. The heuristics may incorrectly treat repetition as formulaic AI behavior even though it is an intentional creative choice.

## Rate Limiting Plan

The `POST /submit` endpoint will use Flask-Limiter.

Chosen limit:

```txt
10 submissions per minute
100 submissions per day
```

Reasoning:

A normal creator on a writing platform might submit several drafts or posts in a short period, so the system should allow a small burst of activity. However, a script sending hundreds of requests per minute could overload the API or abuse the Groq free tier. A limit of `10 per minute` allows normal testing and creator use while blocking rapid spam. A daily limit of `100 per day` is enough for one creator or tester but prevents extreme repeated use.

For local development, Flask-Limiter will use:

```txt
storage_uri="memory://"
```

This is acceptable for the project because the rate limit resets when the local server restarts.

## Stretch Feature Plan

### Stretch 1: Ensemble Detection

This project will include three signals instead of only two:

1. Groq LLM assessment
2. Stylometric heuristics
3. Formulaic/repetition signal

The API response will show individual signal scores alongside the combined ensemble result.

### Stretch 2: Provenance Certificate

The system will include a simple verification step through `POST /verify/<content_id>`. A creator can submit a verification statement for their content. Once verified, the content status can include a verified flag, and the label can change to:

```txt
Verified creator: this creator completed a provenance check for this content.
```

This is distinguishable from the normal likely-human label because it indicates a creator completed an extra verification step.

### Stretch 3: Analytics Dashboard

The system will include `GET /analytics`, and optionally a simple `GET /dashboard` route, to show at least three metrics:

1. Detection pattern: ratio of likely AI, uncertain, and likely human results
2. Appeal rate
3. Average confidence score

These metrics help platform moderators understand how the system is being used.

### Stretch 4: Multi-Modal Support

The system will include `POST /submit-metadata` for non-text or semi-structured content such as image metadata, captions, or creator-provided image descriptions.

Example request:

```json
{
  "creator_id": "creator-123",
  "content_type": "image_metadata",
  "description": "A digital painting of a panda astronaut on a purple planet.",
  "metadata": {
    "tool": "unknown",
    "caption": "A panda astronaut explores PlanetX94."
  }
}
```

The pipeline will not perform real image analysis. Instead, it will process the image description, caption, and metadata fields using adapted text and metadata signals. The README will clearly explain that this is metadata-based multi-modal support, not computer vision.
